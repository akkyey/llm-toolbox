# 開発ロードマップ・機能追加候補 (TODO & Features)

このドキュメントは、LLM監視ツール（stateforge-monitor）をさらに使いやすく堅牢にし、またオープンソースとして公開する際に目玉となるような機能追加やアーキテクチャ改善のアイデアをまとめたものです。

## 🚀 ユーザー体験・機能追加 (User Experience & Features)

### 1. 警告のカラー表示（視覚的なアラート）
コンテキスト量やVRAM使用量が限界に近づいた際、直感的に気づけるように色を変化させる機能。
- **実装アイデア**: 
  - `総記憶: 24572/32768 tok` の割合を計算。
  - 80%超過で黄色（Warning）、95%超過で赤色（Critical / 点滅）。
  - 同様に、VRAM使用率（例: 10GB/12GB）にもカラーリングを適用。
- **メリット**: Neovim等で作業中、視界の端に入るだけで `:AvanteClear` のタイミングが直感的に判断できる。

### 2. コンテキストシフト（要約・切り捨て）の検知
`llama-server` が限界に達し、内部で古いトークンを切り捨てた（Context Shift）ことを検知する機能。
- **実装アイデア**:
  - `journalctl` ログから `context shift` や `removing tokens` 等のパターンをキャプチャ。
  - TUIのステータスに `[Context Shift 発生]` のような警告を一定時間表示。
- **メリット**: AIが過去の指示を急に忘れた際、「コンテキストが削られたからだ」とすぐに原因を特定できる。

### 3. モデル読み込み（ロード）のプログレス表示
サーバー起動時、重いGGUFファイルをVRAMにロードしている間の進捗を表示する機能。
- **実装アイデア**:
  - `llama_model_load` 等のロード進捗ログ（`%` 表示）をパーサーで拾う。
  - TUIのステータスを「モデル読み込み中... 45%」のように更新。
- **メリット**: サーバー再起動直後の「フリーズしているのかロード中なのか分からない」不安を解消できる。

### 4. スロット（並列処理）の稼働状況
`--parallel 2` 等の複数スロット設定を利用するヘビーユーザー向けの拡張。
- **実装アイデア**:
  - `launch_slot` や `all slots are idle` を解析し、現在稼働中のスロット数をトラッキング。
- **メリット**: 複数ファイル同時処理時など、APIサーバーとしてどの程度余力があるかを可視化できる。

### 5. 今日のサマリー・統計情報
開発セッションのモチベーションを上げるための小さな統計情報。
- **実装アイデア**:
  - デーモン側で「今日生成した合計トークン数」や「計測した最高生成速度 (Max t/s)」を累積記録。
  - TUIの隅にさりげなく表示。

## 🛠️ アーキテクチャ・安定性向上 (Architecture & Stability)

先のコード評価で浮き彫りになった、本番運用や汎用化に向けたシステム面の改善案です。

### 6. ハードウェアパスの動的解決 (Dynamic Sysfs/Hwmon Discovery)
ハードウェアモニターのパスはOSの再起動時やデバイス構成変更時に番号が変わる可能性があるため、固定パスからの脱却を図る。
- **実装アイデア**:
  - `/sys/class/drm/card1/device` や `hwmon2`, `hwmon4` などのハードコードを廃止。
  - ディレクトリ内を探索し、`name` ファイルの内容（`amdgpu` や `k10temp` など）から対象デバイスを動的に特定する。
- **メリット**: 異なる環境や再起動後でも安定してメトリクスを取得できる。汎用性が大幅に向上する。

### 7. 同期的処理の完全非同期化 (Non-blocking Subprocesses)
一部のブロッキング呼び出しを解消し、イベントループの健全性を保つ。
- **実装アイデア**:
  - `SystemMetricsCollector.refresh_pids()` 内の `subprocess.run(["pgrep", ...])` を `asyncio.create_subprocess_exec` に置き換える。
- **メリット**: 瞬間的な遅延がなくなり、デーモン全体の並行処理性能とレスポンスが向上する。

### 8. 監視ポートと設定の外部化 (Configurable Ports)
監視対象が `9090`, `9091` とハードコードされている部分を柔軟に設定可能にする。
- **実装アイデア**:
  - コマンドライン引数 `--ports 9090,9091` や設定ファイル（YAML/JSON）でエンジン定義を渡せるようにする。
- **メリット**: ユーザーごとの環境に容易に適合できるようになり、OSSとしての使い勝手が増す。

### 9. 画面サイズ不足時のフォールバック処理 (TUI Responsive Layout)
ターミナルサイズが極端に小さい場合のクラッシュやUI崩れを防ぐ。
- **実装アイデア**:
  - `stdscr.getmaxyx()` で縦横幅をチェックし、描画に必要な最小サイズを下回る場合は「ターミナルを広げてください」という警告のみを表示する。
- **メリット**: 環境に依存しない堅牢でGracefulなTUIが実現できる。
